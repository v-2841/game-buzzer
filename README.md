# 🔔 Buzzer

Игра-«кто первый нажал». Ведущий открывает приём ответов, участники жмут кнопку
на своих телефонах — отвечает тот, кто нажал **раньше всех**, честно, несмотря на
разный пинг. Ведущий ставит очки, в конце — таблица результатов и конфетти лидеру.

Всё работает в реальном времени через WebSocket, состояние игры живёт в памяти
сервера (без базы данных), приложение mobile-first.

---

## Как это работает

1. Ведущий открывает приложение и создаёт игру → получает **код комнаты** (напр. `ELYPY`)
   и QR/ссылку для участников.
2. Участники заходят по ссылке `/r/<код>`, вводят имя.
3. Ведущий жмёт **«Разрешить отвечать»** — у всех загораются кнопки.
4. Участники жмут — сервер собирает почти одновременные нажатия в коротком окне и
   выбирает того, кто нажал раньше по **реальному времени нажатия**, а не по тому,
   чей пакет первым добежал до сервера.
5. Ведущий ставит очки отвечавшему (шкала −3…+3) и открывает следующий раунд.
6. В конце — **«Завершить игру»**: всем показывается финальная таблица с местами,
   лидеру (на его экране) летит конфетти.

### Честность «кто первый» (главная фича)

Наивная схема «победил тот, чей пакет первым пришёл на сервер» нечестна — выигрывает
тот, у кого быстрее интернет. Здесь используется **синхронизация часов** (алгоритм
Кристиана):

- Клиент периодически пингует сервер (`clock:ping`), измеряет RTT и вычисляет
  смещение своих часов относительно серверных.
- При нажатии клиент шлёт время нажатия, приведённое к **серверным часам**.
- Сервер открывает «окно сбора» (адаптивное, 120–700 мс — тем шире, чем выше
  пинги в комнате), собирает все нажатия и выбирает минимальное время нажатия.
- Время нажатия сервер **зажимает** в физически возможный диапазон (нельзя «нажать»
  раньше, чем пакет мог долететь, или в будущем) — это защищает от `NaN`/подделок.

Подробности — в [`server/src/rooms.ts`](server/src/rooms.ts) (`buzz`, `closeCollection`,
`collectWindowFor`) и [`client/src/lib/socket.ts`](client/src/lib/socket.ts) (`ClockSync`).

## Стек

- **Язык:** TypeScript везде (общий пакет типов — единый контракт WebSocket-сообщений).
- **Сервер:** Node 24 + Socket.IO + Express. Состояние комнат — в памяти, без БД.
- **Клиент:** React 19 + Vite + Bootstrap (mobile-first).
- **Монорепо:** pnpm workspaces.
- **Контейнеризация:** Docker (отдельные конфиги для dev и prod).
- **Тесты:** Vitest.

## Запуск

Нужен только **Docker** (Node на хосте не требуется — всё внутри контейнера).

### Разработка (hot-reload)

```bash
docker compose -f docker-compose.dev.yml up
```

- UI: <http://localhost:5173>
- С телефона: `http://<hostname>:5173` (телефон и компьютер в одной Wi-Fi сети).
- Исходники примонтированы — правки в коде подхватываются на лету.

### Прод (локальная проверка прод-сборки)

```bash
docker compose up --build
```

- Приложение: <http://localhost:8010>
- Собирается self-contained образ: Vite-сборка клиента + один бандл сервера, без
  исходников и без `node_modules`. Это то, что разворачивается на реальном сервере.

dev и prod используют разные имена Docker-проектов (`buzzer-dev` / `buzzer-prod`) и
порты, поэтому могут работать одновременно.

### Без Docker

Требуется Node ≥ 20.19 и pnpm. Из корня:

```bash
pnpm install
pnpm dev          # server (:3000) + client (:5173) параллельно
```

## Конечный автомат раунда

```
IDLE ──"разрешить отвечать"──► ARMED ──первое нажатие──► COLLECTING
  ▲                              ▲                            │
  │                              │      окно закрылось,       ▼
  └──── resetScores ◄── ARMED ◄──┴── начисление очков ◄── ANSWERING
                                         (только победителю)

любая фаза ──"завершить игру"──► FINISHED ──"новая игра"──► IDLE
```

Сервер хранит только состояние комнаты (фаза, игроки, очки, история начислений —
её ведущий может править и удалять записи). Понятия «вопрос» у сервера нет —
раундами управляет ведущий.

## Безопасность и ограничения

Все входящие сообщения проверяются на сервере (типы, конечность чисел), действия
ведущего привязаны к admin-токену, есть базовая защита от абуза:

- лимит комнат и игроков в комнате, rate-limit на сокет-события;
- время нажатия и клиентский RTT зажимаются в доверенный диапазон;
- состав раунда фиксируется в момент `arm()` — поздно зашедший не влезет в текущий сбор;
- пустые комнаты автоматически вычищаются через ~5 минут простоя.

Игра рассчитана на дружескую компанию: при клиентской синхронизации часов
абсолютную защиту от читерства гарантировать нельзя, но грубые подделки и сбои
обезврежены, а порядок определяется честно.
